Six-layer architecture

Mirror's LLM pillar is a layered stack where each layer has a precise, isolated role. The model is swappable at Layer 4 without touching anything above it. Click any layer to expand.

Layer 1 — Application interface// what the user touches
›
Receives raw user input — text, voice, session context — and packages it into a structured request object. Attaches the user's cognitive profile (from previous sessions), the active module (Canvas, X-ray, Devil's Advocate, etc.), and session metadata before passing it downstream to the orchestrator. Also receives the final response and renders it in Mirror's visual output formats — radar charts, DNA strips, decision maps. This layer never calls an LLM directly.
Session context builderProfile injectorVisual rendererModule state manager
Layer 2 — Orchestrator// the brain — routes, decides, delegates
›
The most critical layer. Reads the incoming request and makes three decisions: (1) which model to use based on task type and complexity, (2) single-shot or multi-step agentic workflow, and (3) which RAG context to retrieve. Does not generate text — it directs. Implemented as a lightweight fast-model call (GPT-4o-mini or Gemini Flash) that returns a JSON routing directive in ~200ms.
Task classifierModel routerAgent spawnerRAG trigger~200ms target
Layer 3 — Reasoning engine// deep thinking, used surgically
›
Not every request needs deep reasoning. When the orchestrator detects a complex task — multi-session pattern synthesis, bias archaeology, weekly report generation — it routes here. Uses a model with extended thinking: o3, o4-mini, or Claude 3.7 Sonnet with extended thinking enabled. Slower (10–30s) and more expensive — used surgically, never for conversational turns. Outputs a structured JSON plan, not a user-facing response.
Extended thinking ONChain-of-thoughtMulti-step synthesisAsync executionOutputs JSON plan
Layer 4 — Execution model// fast, conversational, always-on — swappable
›
Handles all real-time conversational turns. Needs to be fast (<2s first token), high quality, and strongly instruction-following. This is where swappability lives. Mirror's ModelAdapter abstraction means you swap Claude Sonnet, GPT-4o, Gemini 1.5 Pro, or Llama 3.1 70B by changing one config value — zero code changes above this layer. Receives: system prompt + user message + RAG chunks (8–12K tokens) + user profile summary + last 3 turns.
Swappable via ModelAdapterStreaming outputTool-use enabledTarget <2s first token
Layer 5 — RAG knowledge layer// research papers + user cognitive history
›
Two parallel vector stores. Store A — metacognition research corpus (200–500 papers, chunked at 512 tokens, 64-token overlap, embedded with text-embedding-3-large). Store B — user's cognitive history (logged decisions, surfaced patterns, session summaries, prediction outcomes). Both queried on every request. Top-5 research + top-3 history moments retrieved, re-ranked, injected as structured XML blocks.
Research store (static, shared)User history store (per-user)Pinecone / pgvectorHybrid dense + sparse retrieval
Layer 6 — Memory & persistence// what Mirror remembers across sessions
›
Three tiers. Short-term: current session in context. Mid-term: compressed session summary (300–500 tokens) written at close, stored in user's vector store. Long-term: structured cognitive profile JSON updated each session — dominant patterns, calibration score, bias fingerprint, tracked beliefs, growth timeline. Injected at the start of every new session (~2K tokens). This 2K JSON is the distillation of everything Mirror has ever learned about the user.
In-context (session)Vector store (mid-term)Profile JSON (long-term)Auto-compressed summaries
The abstraction principle
Every layer communicates only with the layer directly below it through a defined interface. The execution model is one implementation behind a ModelAdapter interface. Swap the implementation — the interface stays identical. This is the architectural decision that makes Mirror future-proof against model changes.

How swappability works
All calls go through a ModelAdapter class that normalises API differences between providers. To swap from Claude to GPT-4o, change one value in config.yaml. The adapter handles streaming normalisation, retry logic, token counting, and cost logging. LiteLLM is the recommended open-source library for this — it supports 100+ providers with a unified interface.

04

Agentic orchestration loop

How Mirror's weekly deep review actually runs — a 5-agent sequential pipeline triggered by a single user request. Each agent is a separate LLM call with a specific role.

Trigger
User requests weekly review
Orchestrator classifies as multi-step agentic task. Spawns supervisor agent. User sees "Mirror is preparing your review..." state. Pipeline begins async.
Step 1 — Retrieval agent
Pulls all context needed for synthesis
Fetches last 7 days of session transcripts and summaries from user's vector store. Retrieves current cognitive profile JSON. Queries research RAG for patterns relevant to themes surfaced this week.
vector_search(user_sessions, days=7)fetch_profile(user_id)rag_query(themes[])
Step 2 — Reasoning agent (o3 / Claude 3.7 extended thinking)
Deep synthesis — the expensive call
Receives all retrieved data. Runs extended thinking: cross-session pattern detection, belief update identification, calibration drift scoring, single strongest metacognitive signal extraction. Takes 10–25 seconds. Runs fully async.
extended_thinking: ONbudget_tokens: 8000output: structured_analysis
Step 3 — Planning agent
Converts reasoning output into a structured session plan
Takes the reasoning output. Creates a typed JSON plan: 5 personalised reflection questions in order, which module to open first, priority pattern for the week, one research citation to ground the session.
create_session_plan(analysis)output: SessionPlan JSON
Step 4 — Execution model (Sonnet / fast model)
Generates the user-facing weekly review
Receives the plan JSON as context. Generates the actual weekly review — written in Mirror's voice, research-grounded, personalised to this user's exact patterns. Streamed directly to the UI.
stream: truesystem_prompt: Mirror_v2context: plan + profile
Step 5 — Memory write-back
Closes the loop, prepares for next session
Session compressed to ~400 tokens and written to user's vector store. Cognitive profile JSON updated: new dominant patterns, revised calibration score, growth timeline entry. Ready for next retrieval.
write_summary(session_id)update_profile(user_id, delta)index_moments()
Framework recommendation
LangGraph is recommended for Mirror's agentic layer — it handles stateful multi-agent graphs with conditional routing, which maps precisely to this pipeline. CrewAI is simpler to start with but less flexible for complex conditional flows. A custom lightweight orchestrator (plain Python with async/await) is viable for early-stage and gives maximum control.
05

RAG-based planning

How Mirror turns research papers + personal history into a personalised, grounded response — on every single call. The plan step is what separates Mirror from a standard RAG chatbot.

Input
Raw user message
Text + session ID + active module + last 3 turns compressed to summaries
Parse
Intent extraction
Fast classifier extracts: topic tags, emotional valence, decision type, cognitive load signal, hinted biases
Retrieve A
Research corpus query
Top-5 chunks from metacognition papers. Re-ranked by cosine similarity + cross-encoder re-ranker
Retrieve B
User history query
Top-3 past session moments where this same pattern appeared — enables "this is the same thing from 3 weeks ago"
Plan
Micro-plan creation
Reasoning model creates 3-sentence plan: pattern to surface, question to ask, citation to use. Structured JSON.
Execute
Execution model generates
Plan injected as context. Execution model writes Mirror's actual response — grounded, cited, personalised. Streamed.
Why the plan step changes everything
Without a plan step, the LLM free-generates — it might surface the right pattern or it might not. With the plan step, the reasoning model has already decided what to say before the fast model says it. The execution model becomes a fluent writer executing a precise brief. This is how Mirror maintains research grounding and consistency at scale — not through luck or prompt length, but through structured pre-generation planning.
RAG chunk strategy for metacognition papers
Chunk size: 512 tokens with 64-token overlap. Embedding: text-embedding-3-large or embed-english-v3.0. Index: HNSW for fast approximate nearest-neighbour. Re-ranking: cross-encoder on top-20 before selecting top-5. Metadata per chunk: author, year, concept_tags[], bias_categories[] — used for filtered retrieval when orchestrator already knows the relevant bias category from the user's profile.

The rule: selective injection beats brute-force
Never stuff the full history into context. Mirror uses compressed session summaries (not full transcripts) for sessions older than 2 weeks, full transcripts only for the last 3 sessions, and targeted retrieval for pattern continuity. This keeps every call under 30K tokens — fast, cheap, precise — while retaining the depth of a year of thinking history. The cognitive profile JSON does the heavy lifting: it is a 2K-token distillation of everything Mirror has ever learned about the user.
